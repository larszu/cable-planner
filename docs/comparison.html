3 · Detail-Vergleich pro Tool

Referenz Cable-Planner (dieses Projekt)

Electron-App mit React-Renderer und Node-Main-Prozess. Module nach Domäne segmentiert (Canvas, Library, Properties, Rack, Atem, Patch, Export, Import). Zustand mit Persist-Middleware, eigener History-Store mit Transactions im draw.io-Stil. ReactFlow für 2D-Canvas, Three.js für 3D-Racks. Atomic File-Writes (tmp → rename + .bak). IPC über contextBridge mit getypten Channels pro Domäne.

Vorteile

  • Plattform-übergreifend (Win/macOS/Linux) durch Electron
  • Offenes JSON-Format → Diff-bar, Git-tauglich, kein Vendor-Lock-in
  • Domäne tief: AV-Connector-Typen, Multi-Layer-Auto-Detect, Patchfeld-Modell, ATEM-Live-Control
  • 3D-Rack-View und STL-Export — selten in AV-Tools
  • Atomic File-Writes mit .bak — robust gegen Crash-Mid-Write
  • Undo/Redo mit Transaction-Pattern (draw.io-Vorbild)
  • Mobile-Share-View für Aufbau-Team auf der Baustelle
  • Modul-Layout sauber: 19 Komponenten-Subdomänen, klar getrennte IPC-Layer
  • Keine Lizenzkosten, kein Cloud-Zwang, volle Datenhoheit

Nachteile

  • Electron-Overhead: ~150–250 MB RAM Baseline, ~120 MB Installer
  • Eine Person als Maintainer → Bus-Faktor 1
  • Kein Plugin-/Skript-API → Erweiterungen brauchen Code-Fork
  • Keine kollaborative Echtzeit-Bearbeitung (Single-User-Datei)
  • Keine zertifizierte Symbolbibliothek (IEC/VDE-konform) wie EPlan
  • Renderer-Code 2.6 MB src/ — Wachstum ohne klare Modulgrenzen-Verträge riskant
  • Kein integriertes Cloud-Sync (nur Mobile-Share-Hop)
  • Mobile ist Lese-/Share-View, keine vollwertige Bearbeitung

Direkte Konkurrenz WireCAD (Holbrook Enterprises)

Etablierter Markt-Standard für Broadcast-/AV-Verkabelung seit ~2003. WPF/.NET-Desktop-App mit SQLite-Backend. Bietet One-Line-/Multi-Line-Schaltbilder, automatische Kabelnummerierung, Lagebücher, sehr umfangreiche Symbol-Datenbank. Zielgruppe: Systemintegratoren und Broadcaster.

Vorteile

  • Sehr große, gepflegte Symbol-Bibliothek (Tausende Geräte vom Hersteller)
  • Reife Auto-Routing-Engine
  • BOM mit Hersteller-Preisen, Procurement-tauglich
  • SDK für eigene Reports/Plugins
  • Etablierter Support, Schulungen

Nachteile

  • Nur Windows
  • Hohe Einstiegskosten (~$1500–4500)
  • UI-Design altbacken (Toolbar-Klötze, schmale Dialoge)
  • Proprietäres Datenformat — Vendor-Lock-in
  • Keine 3D-Rack-Visualisierung
  • Keine direkte Hardware-Kontrolle
  • Keine mobile View

Direkte Konkurrenz D-Tools System Integrator

System-Integrator-Tool mit Fokus auf den ganzen Projekt-Lifecycle (Angebot → Plan → Procurement → Inbetriebnahme → Service). Visio-basiert für Pläne, eigene Datenbank für Bauteile/Preise. SQL-Server-Backend. Zielgruppe: Custom-Install-Firmen.

Vorteile

  • Geschäftsprozess komplett abgedeckt (Quote → Invoice)
  • Sehr starke Hersteller-Datenbank mit Live-Preisen
  • Mobile-App für Außendienst/Service
  • Multi-User mit zentralem Server
  • Integration zu QuickBooks, Salesforce

Nachteile

  • Sehr teuer (mehrere $1000/Jahr je User)
  • Windows + SQL Server nötig
  • Plan-Editor lehnt sich an Visio an — Cable-Routing schwächer als WireCAD
  • Sehr steile Lernkurve
  • Overkill für kleine Studios/Freelancer
  • Kein Open-Format

Industrie-Elektrik EPlan Electric P8 / WSCAD ELECTRIX

Marktführer für Schaltschrank- und Industrie-Elektrik-Planung. Native Desktop-Apps mit umfangreichen Bibliotheken, 3D-Cabinet-Builder, IEC-konformen Symbolen, ERP-Anbindung (SAP). EPlan ist Standard im Maschinenbau, WSCAD die günstigere Alternative.

Vorteile

  • Zertifizierte Symbolbibliotheken (IEC, NFPA, GOST)
  • 3D-Schaltschrank-Builder mit Bohrbild-Generierung
  • ERP/PDM-Integration (SAP, Teamcenter)
  • Skripting-API (.NET)
  • Klemmenplan, Aderverfolgung, Revisionsmanagement
  • Multi-Sprachen-Pläne automatisch

Nachteile

  • Sehr teuer (5–15k €/Lizenz, +Module)
  • Nur Windows
  • Steile Lernkurve, Schulung nötig
  • Falsche Domäne für AV/Broadcast: zu viel Industrie-Ballast (Klemmen, SPS)
  • Datenformate proprietär und binär
  • Performance bei großen Projekten zäh

Allgemein draw.io / diagrams.net

Apache-2.0-lizenzierter Diagramm-Editor (Web + Desktop-Variante). Basiert auf mxGraph (das Vorbild für Cable-Planners Transaction-Undo-Pattern). Sehr beliebt für Architektur-/Flussdiagramme, weniger geeignet für domänenspezifische Verkabelung.

Vorteile

  • Komplett kostenlos, Open Source
  • Läuft im Browser, kein Install nötig
  • Sehr reife Engine (mxGraph), saubere Undo/Redo
  • Export in vielen Formaten (SVG, PDF, PNG, XML)
  • Cloud-Sync optional (Google Drive, OneDrive)
  • Großer Community-Pool von Shapes

Nachteile

  • Generisch — keine AV-/Broadcast-Domäne
  • Keine Auto-Kabel-Nummerierung, kein BOM
  • Keine Layer-Auto-Detect aus Connector-Typ
  • Kein 3D-View
  • Keine Hardware-Integration
  • Keine Patchfeld-Logik (Eingang/Ausgang-Mapping)

Allgemein Microsoft Visio

Klassiker für Business-Diagramme, Netzwerk- und Anlagenpläne. Native Win32-/COM-App mit VBA-Scripting. Heute auch Web-Variante in MS 365. Nicht primär für AV — wird aber oft als Notbehelf eingesetzt.

Vorteile

  • Etablierter Standard in Unternehmen
  • Sehr ausgereifte Konnektor-Logik
  • VBA/COM-Scripting für Automatisierung
  • Stencil-Marktplatz mit AV-Symbolen
  • Office-Integration (Excel-Daten-Binding)

Nachteile

  • Nur Windows (Desktop) — macOS-Nutzer müssen auf Web
  • Abo-Modell (Microsoft 365)
  • Generisch — keine AV-/Broadcast-Domäne out of the box
  • VSDX-Format zwar offen, aber Tools-Ökosystem klein
  • Kein 3D-View
  • Kein nativer BOM-Export

Open Source · Browser-First tldraw / Excalidraw

Moderne Web-Canvas-Tools mit Fokus auf UX. tldraw bietet SDK zum Embedden, Excalidraw setzt auf Hand-gezeichnete Ästhetik. Beide React-basiert, beide Open Source. Architektonisch das nächste Vorbild für Cable-Planners Renderer.

Vorteile

  • Sehr moderne UX (smooth pan/zoom, Multi-Touch)
  • Browser-First → instant share via URL
  • SDK / Library-Mode für Eigenbau-Apps
  • Saubere TypeScript-APIs
  • Echtzeit-Kollaboration (tldraw Sync, Excalidraw+)

Nachteile

  • Generische Whiteboards, keine Domäne
  • Keine Hardware-Integration (Browser-Sandbox-Grenzen)
  • Kein nativer File-Zugriff ohne Cloud/User-Action
  • Keine 3D-View
  • Keine ATEM/USB/Bonjour-Integration möglich

4 · Strukturelle Stärken & Risiken des Cable-Planner

4.1 · Was strukturell gut gelöst ist

  • Klare Schichten: main/ipc, main/services, main/util vs. renderer/components, renderer/store, renderer/lib — die Trennung ist konsistent.
  • Domänen-orientierte IPC-Channels: project:*, library:*, atem:* statt einem mega-Handler. Erleichtert Erweiterungen.
  • Atomic Writes mit util/atomicWrite.ts als shared util — gleicher Schutz für Projekt-JSON und Library-JSON.
  • Transaction-basiertes Undo (draw.io-Pattern) + 200ms-Coalesce — UX-Niveau, das die meisten kommerziellen AV-Tools nicht haben.
  • Layer-Auto-Detect aus Connector-Typ (BNC→Video, XLR→Audio, …) — domain-smart Default statt manueller Klassifikation.
  • Heal-Funktion beim Projekt-Laden (healProjectPositions) — gracefully Migration alter Dateien, kein Bruch durch Schema-Evolution.
  • Mobile-Share-View als HTTP-Hop direkt aus der App — kein Backend-Server nötig.

4.2 · Wo strukturelle Risiken liegen

  • Renderer-Größe (2.6 MB src/, 19 Komponenten-Subdomänen): ohne klare Schnittstellen-Verträge zwischen Subdomänen können Imports kreuz und quer wachsen. Empfehlung: eine ESLint-Regel für allowed-imports pro Subdomäne (z. B. eslint-plugin-boundaries) oder ein explizites renderer/api-Layer als Single Source of Truth.
  • Store-Monolith (adressiert ✓ #308): projectStore.ts wurde inzwischen via Slice-Pattern auf 14 Slices unter store/slices/ aufgeteilt — der Haupt-Store ist von 2178 auf ~1146 LOC geschrumpft.
  • Bus-Faktor 1 (adressiert ✓): docs/architecture.md hält die nicht-verhandelbaren Invarianten (Schema-Migration, IPC-Channels, Heal-Logic, Atomic Writes, Slice-Komposition, i18n-Konventionen).
  • Keine Plugin-API: Erweiterungen brauchen Code-Forks. Wenn Drittnutzer eigene Connector-Typen, Reports oder Hardware-Integrationen brauchen, müssten sie patchen. Eine schmale Plugin-API (z. B. nur Reports und Library-Loader) wäre eine günstige Investition.
  • Keine kollaborative Bearbeitung: Single-File-Modell. Wenn zwei Teams parallel am Plan arbeiten wollen, brauchen sie Merge per Hand. CRDT (z. B. Yjs) wäre architektonisch addable, aber nicht trivial.
  • Mobile als zweiter Renderer ist eine Read-View. Wenn Mobile-Edit kommen soll, braucht's eine ernsthaftere API-Schicht statt nur Static-File-Share.
  • Testing-Lücke: keine Test-Skripte im package.json. Bei 61.5k LOC ist das ein wachsendes Regressions-Risiko — gerade bei Undo/Redo und Heal-Logic. Empfehlung: vitest + 30–50 Snapshot-Tests auf Projekt-Migration / Heal / History.

4.3 · Wo Cable-Planner strukturell besser ist als die Konkurrenz

  • Plattform-Reichweite: einzige der Liste, die Win/macOS/Linux nativ liefert (außer Browser-only-Tools).
  • Offenes Dateiformat: JSON ist diffbar/git-tauglich. WireCAD/D-Tools/EPlan sind Black-Box.
  • Tech-Stack-Modernität: React 19 + TS 6 + Vite 8. WireCAD/EPlan sitzen auf 10+ Jahre alten .NET-Stacks.
  • 3D + 2D in einem Tool: Three.js-Rack-View + ReactFlow-Canvas. Kein Konkurrent auf der Liste hat beides nativ.
  • Live-Hardware-Integration: ATEM-Steuerung direkt aus der Plan-App. Weder WireCAD noch D-Tools können das.
  • Undo-UX: Transaction + Coalesce auf Niveau draw.io/mxGraph. Bei WireCAD/Visio ist Undo deutlich klobiger.

5 · Fazit & Positionierung

Architektonisch sitzt Cable-Planner in einer Lücke: modernes Tech-Stack-Niveau wie tldraw/draw.io kombiniert mit Domänen-Tiefe wie WireCAD und Live-Hardware-Integration, die keiner der Konkurrenten anbietet. Bus-Faktor und Store-Monolith sind inzwischen adressiert (architecture.md, Slice-Pattern in store/slices/). Fehlende Plugin-API und Tests bleiben als günstige Investitionen offen.

Klassische Positionierung gegenüber den Wettbewerbern:

  • vs. WireCAD — Cable-Planner gewinnt bei Plattform-Support, Format-Offenheit, UX-Modernität, Hardware-Integration. WireCAD gewinnt bei Symbolbibliothek-Tiefe und etabliertem Workflow.
  • vs. D-Tools — Andere Liga: D-Tools ist Business-Lifecycle, Cable-Planner ist Plan-Tool. Komplementär statt konkurrierend.
  • vs. EPlan/WSCAD — Falsche Domäne — EPlan ist Industrie-Elektrik, Cable-Planner ist AV. Kein direkter Wettbewerb.
  • vs. draw.io/Visio — Cable-Planner gewinnt klar bei Domänen-Smart-Defaults (Connector-Layer, BOM, Patch). Allgemeine Tools bleiben Notbehelf für AV.
  • vs. tldraw/Excalidraw — Andere Liga: Whiteboard vs. Fach-Tool. tldraw könnte aber als Render-Engine-Inspiration dienen, falls ReactFlow irgendwann zu eng wird.
